## The Problem

When an incident happens, the first question is always **"what process did
this?"**  The standard UNIX toolbox falls short in predictable ways:

| Tool | Where it fails in practice |
|------|---------------------------|
| `lsof` | Point-in-time snapshot — a script that opens, writes, and closes a file in milliseconds leaves no trace |
| `inotifywait` | Tells you *a* file changed, but **has no access to the PID** — kernel inotify simply doesn't carry it |
| `auditd` | Technically correct, but requires a daemon, a rule syntax that most operators have to look up, verbose JSON/text logs, and measurable overhead on write-heavy systems |

The gap is acute during incident response: you are watching a config file get
corrupted in real time, you know *when* it happens, but you have no fast way
to attribute the action to a process.

---

## The Solution

`whodid` is a **single statically-linked C binary** (~30 KB) that closes this
gap using the Linux **`fanotify`** kernel API:

* **PID on every event** — `fanotify` delivers the PID of the acting process
  in the event metadata itself; `whodid` resolves it to a full executable
  path via `/proc/<pid>/exe`.
* **No daemon, no config** — one command, one argument:
  `sudo whodid /path/to/watch`
* **Kernel-level interception** — events are generated by the VFS layer,
  before any user-space hook can suppress them.
* **Pipeline-friendly output** — plain text columns, grep/awk/tee ready,
  colour auto-disabled when piped.
* **Native syslog/journald integration** — pass `-l` / `--syslog` and every
  event lands in the system journal alongside the rest of your audit trail.
* **Hardened build** — compiled with `-fstack-protector-strong`,
  `-D_FORTIFY_SOURCE=2`, full RELRO, and PIE; all output strings are
  sanitised against terminal escape-sequence injection.

## Requirements

| Requirement | Notes |
|-------------|-------|
| Linux kernel ≥ 4.0 | Basic mode — WRITE / MODIFY events |
| Linux kernel ≥ 5.1 | Full mode — + CREATE / DELETE / RENAME / ATTRIB |
| `root` or `CAP_SYS_ADMIN` | Required by `fanotify_init(2)` |
| `gcc`, `make` | Build-time only — no runtime dependencies |

**Debian / Ubuntu**: kernel 5.10+ (Bullseye) or 6.1+ (Bookworm) → full mode.  
**RHEL / Rocky / Alma 9**: kernel 5.14 → full mode.  
**WSL2**: run `wsl --update` from Windows PowerShell to get a kernel ≥ 5.1.

## Installation

### Pre-built packages (recommended)

Every [GitHub Release](../../releases/latest) ships ready-to-install packages
built by CI — no compiler needed.  Each artifact is GPG-signed and
SHA-256 checksummed.  See [**Verifying signatures**](#verifying-signatures)
below.

| Package | Target distribution | Install command |
|---------|---------------------|-----------------|
| `whodid_<ver>_<arch>.deb` | Debian, Ubuntu, Raspbian | `sudo dpkg -i whodid_*.deb` |
| `whodid-<ver>-1.<arch>.rpm` | Fedora, RHEL, CentOS, openSUSE | `sudo dnf install whodid-*.rpm` |
| `whodid-<ver>-<arch>.AppImage` | Any modern Linux (no install) | `chmod +x whodid-*.AppImage && sudo ./whodid-*.AppImage /path` |
| `PKGBUILD` + `.SRCINFO` | Arch Linux (AUR) | see [AUR section](#aur-package-arch-linux) below |

### Verifying signatures

All packages attached to a release are signed with a detached GPG signature
(`.asc` file) and covered by the `SHA256SUMS` manifest.

```bash
# Download the release assets (replace <version> and <arch> as appropriate)
curl -sSLO https://github.com/0whis/whodid/releases/latest/download/whodid_<version>_<arch>.deb
curl -sSLO https://github.com/0whis/whodid/releases/latest/download/whodid_<version>_<arch>.deb.asc
curl -sSLO https://github.com/0whis/whodid/releases/latest/download/SHA256SUMS
curl -sSLO https://github.com/0whis/whodid/releases/latest/download/signing-key.pub.asc

# Verify (uses verify.sh from the repository)
chmod +x verify.sh
./verify.sh --key signing-key.pub.asc whodid_<version>_<arch>.deb
```

Or verify manually with GPG and sha256sum:

```bash
gpg --import signing-key.pub.asc
gpg --verify whodid_<version>_<arch>.deb.asc whodid_<version>_<arch>.deb
sha256sum --check --ignore-missing SHA256SUMS
```

### Quick install (Debian / Ubuntu / WSL2 — build from source)

```bash
git clone https://github.com/0whis/whodid.git
cd whodid
chmod +x install.sh
sudo ./install.sh
```

The script checks and installs build dependencies (`gcc`, `make`, `libc6-dev`),
verifies the kernel version, compiles the binary, and copies it to
`/usr/local/bin/whodid`.

### .deb package (Debian / Ubuntu)

**Option A — download the pre-built package** (see [Pre-built packages](#pre-built-packages-recommended) above).

**Option B — build locally from source:**

```bash

cd whodid
./build-deb.sh                          # produces whodid_1.0.0_<arch>.deb
sudo dpkg -i whodid_1.0.0_*.deb
```

```bash
sudo dpkg -i whodid_1.0.0_*.deb
```

```bash
sudo dpkg -P whodid        # purge (same as -r for this package)
```

## Usage

```
Usage: whodid [OPTIONS] <path>

  Real-time file-system activity monitor.
  Shows which process (PID + executable) is accessing or modifying
  files under <path>.

  Requires root or CAP_SYS_ADMIN.

Options:
  -a, --all       Show all events: OPEN, READ, CLOSE (adds noise)
                  Default: WRITE, CREATE, DELETE, RENAME, ATTRIB only
  -l, --syslog    Also write each event to the system journal via syslog(3)
                  View with: journalctl -t whodid
  -n, --no-color  Disable ANSI colors (auto-off when output is not a tty)
  -s, --self      Include events generated by whodid itself
  -q, --quiet     Suppress the startup banner
  -v, --version   Print version and exit
  -h, --help      Show this help and exit
```

### Examples

```bash
# Watch /etc/ for writes, creates and deletes
sudo whodid /etc/

# Watch /var/log/ including open/read events
sudo whodid -a /var/log/

# Tee to a log file (colours disabled automatically)
sudo whodid /home/ | tee activity.log

# Minimal output — suitable for scripts or log pipelines
sudo whodid -q -n /tmp/

# Send to stdout AND to the system journal simultaneously
sudo whodid --syslog /etc/
journalctl -t whodid -f

# Filter events to a single process name
sudo whodid /var/www/ | grep nginx

# Hunt for persistence: watch common drop zones in the background
sudo whodid /etc/cron.d/ &
sudo whodid /etc/systemd/system/ &
```

## Incident Response Examples

### Monitor `/etc/` for unauthorized modifications

The classic incident-response use case — run as soon as a host is suspected
of being tampered with:

```bash
# Start monitoring in the foreground
sudo whodid /etc/

# In a separate terminal, watch the live journal stream
journalctl -t whodid -f
```

Sample output when a script silently modifies `sshd_config`:

```
15:42:07 | 4821    | root         | /tmp/.hidden/installer          | WRITE     | /etc/ssh/sshd_config
15:42:08 | 4821    | root         | /tmp/.hidden/installer          | ATTRIB    | /etc/ssh/sshd_config
```

### Hunt for persistence mechanisms

Watch all common drop-zones for new files simultaneously:

```bash
sudo whodid /etc/cron.d/ &
sudo whodid /etc/cron.daily/ &
sudo whodid /etc/systemd/system/ &
sudo whodid /etc/profile.d/ &
sudo whodid /etc/ld.so.conf.d/ &
```

### Record events to a file and query them later

```bash
# Log everything to a file with no colour or banner
sudo whodid -q -n --syslog /var/www/ | tee /var/log/whodid-www.log

# Query the system journal for DELETE events from the last hour
journalctl -t whodid --since "1 hour ago" | grep DELETE

# Search the plain-text log for a specific process
grep 'nginx' /var/log/whodid-www.log

# Find non-root actors modifying sensitive paths
grep 'whodid' /var/log/syslog | grep -v 'user=root'
```

### Filter syslog output for specific patterns

When `--syslog` is active, each event is written in key=value format that is
easy to parse:

```
whodid[1234]: action=WRITE pid=9871 user=deploy process=/usr/bin/rsync file=/var/www/html/index.php
```

Useful one-liners:

```bash
# All events involving a DELETE action today
journalctl -t whodid --since today | grep 'action=DELETE'

# All events NOT from a known process
journalctl -t whodid -f | grep -v 'process=/usr/bin/rsync'

# Alert on any write to /etc/sudoers (requires --syslog to be active)
journalctl -t whodid -f | grep 'file=/etc/sudoers'
```

## Output format

```
HH:MM:SS | PID     | PROCESS                                       | ACTION     | FILE
```

| Action | Meaning | Terminal colour |
|--------|---------|-----------------|
| `CREATE` | File or directory created | bright green |
| `DELETE` | File or directory deleted | bright red |
| `MOVE_FROM` | Rename — source name | bright red |
| `MOVE_TO` | Rename — destination name | bright green |
| `MODIFY` | File contents changed (mid-write) | red |
| `WRITE` | File closed after write | red |
| `ATTRIB` | Permissions / ownership changed | yellow |
| `OPEN` | File opened (`-a` only) | dim |
| `READ` | File read (`-a` only) | dim |
| `CLOSE` | File closed without write (`-a` only) | dim |

## How it works

1. **fanotify** — `whodid` calls `fanotify_init(FAN_REPORT_DFID_NAME, …)` to
   open a notification group that embeds file-system object identifiers
   (FIDs) in every event.
2. **Path resolution** — for each event, the directory FID is turned into an
   open file descriptor via `open_by_handle_at(2)`, then the absolute path
   is read from `/proc/self/fd/<n>`.  The file name from the event info
   record is appended.
3. **PID → process** — `readlink("/proc/<pid>/exe")` gives the full
   executable path of the responsible process.  Falls back to
   `/proc/<pid>/comm` if the process has already exited.
4. **Output sanitisation** — all strings are stripped of C0 control
   characters (including `ESC \x1b`) before printing to prevent terminal
   escape-sequence injection via crafted file names.

## Security notes

* `whodid` must be started as **root** (or with `CAP_SYS_ADMIN`) because
  `fanotify_init(2)` requires that capability.  Immediately after acquiring
  the fanotify file descriptor and completing all privileged setup, `whodid`
  drops elevated privileges:
  * The real, effective, and saved UIDs/GIDs are changed to the invoking
    user's IDs (taken from `SUDO_UID` / `SUDO_GID`), or to `nobody`
    (uid/gid 65534) when running as root without `sudo`.
  * In **modern mode** (kernel ≥ 5.1), `CAP_DAC_READ_SEARCH` is retained
    because `open_by_handle_at(2)` requires it.  All other capabilities are
    dropped.
  * In **basic mode** (kernel < 5.1), all extra capabilities are dropped
    entirely.
  * `PR_SET_NO_NEW_PRIVS` is set to prevent any child process from
    re-acquiring privileges via `execve(2)`.
* All path and process-name strings are sanitised before output to prevent
  terminal injection via crafted file names or symlink attacks.
* The binary is compiled with `-fstack-protector-strong`, `-D_FORTIFY_SOURCE=2`,
  full RELRO (`-Wl,-z,relro -Wl,-z,now`), and PIE (`-fPIE -pie`).
* Events from `whodid` itself are suppressed by default (use `-s` to include
  them).

## WSL2 notes

`fanotify` works in WSL2 with a kernel ≥ 5.1.  Run `wsl --update` in Windows
PowerShell to update the WSL2 kernel.  WSL1 is **not** supported — it does
not implement `fanotify`.
