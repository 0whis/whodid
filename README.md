# whodid

> **Real-time file-system activity monitor** — see exactly which process is
> touching your files, right now.

```
15:42:07 | PID: 23901  | PROCESS: /tmp/malware.sh                            | ACTION: WRITE     | FILE: /etc/nginx.conf
15:42:09 | PID: 112    | PROCESS: /sbin/init                                 | ACTION: CREATE    | FILE: /etc/cron.d/backdoor
15:42:11 | PID: 8841   | PROCESS: /usr/bin/vim                               | ACTION: WRITE     | FILE: /etc/hosts
```

---

## Why whodid?

| Tool | Gap |
|------|-----|
| `lsof` | Snapshot only — misses fast open/close cycles |
| `inotifywait` | Detects changes but **cannot report the PID** |
| `auditd` | Correct, but heavyweight daemon + complex rules |
| **whodid** | Lightweight single binary, root-only, instant PID + path |

`whodid` uses the Linux **fanotify** API to intercept file-system events at
the kernel level.  Every event includes the PID of the responsible process,
which is resolved to its full executable path via `/proc/<pid>/exe`.

---

## Requirements

| Requirement | Notes |
|-------------|-------|
| Linux kernel ≥ 4.0 | Basic mode (WRITE events) |
| Linux kernel ≥ 5.1 | Full mode (+ CREATE / DELETE / RENAME) |
| `root` or `CAP_SYS_ADMIN` | Required by fanotify |
| `gcc`, `make` | Build dependencies only |

**Debian / Ubuntu**: kernel 5.10+ (Bullseye) or 6.1+ (Bookworm) → full mode.  
**WSL2**: run `wsl --update` from Windows PowerShell to get a recent kernel.

## Installation

### Quick install (Debian / Ubuntu / WSL2)

```bash
git clone https://github.com/0whis/whodid.git
cd whodid
chmod +x install.sh
sudo ./install.sh
```

### Manual build

```bash
make
sudo make install          # installs to /usr/local/bin/whodid
```

### Uninstall

```bash
sudo ./install.sh --uninstall
# or
sudo make uninstall
```

## Usage

```
Usage: whodid [OPTIONS] <path>

  Real-time file-system activity monitor.
  Shows which process (PID + executable) is accessing or modifying
  files under <path>.

  Requires root or CAP_SYS_ADMIN.

Options:
  -a, --all       Show all events: OPEN, READ, CLOSE (adds noise)
                  Default: WRITE, CREATE, DELETE, RENAME, ATTRIB only
  -n, --no-color  Disable ANSI colors (auto-off when output is not a tty)
  -s, --self      Include events generated by whodid itself
  -q, --quiet     Suppress the startup banner
  -v, --version   Print version and exit
  -h, --help      Show this help and exit
```

### Examples

```bash
# Watch /etc/ for writes, creates and deletes
sudo whodid /etc/

# Watch /var/log/ including open/read events
sudo whodid -a /var/log/

# Log to file (colours disabled automatically)
sudo whodid /home/ | tee activity.log

# Minimal output — suitable for scripts or log pipelines
sudo whodid -q -n /tmp/

# Show version
whodid --version
```

## Output format

```
HH:MM:SS | PID: <pid>  | PROCESS: <executable-path>  | ACTION: <event>  | FILE: <abs-path>
```

| Action | Meaning | Colour |
|--------|---------|--------|
| `CREATE` | File or directory created | bright green |
| `DELETE` | File or directory deleted | bright red |
| `MOVE_FROM` | Rename source | bright red |
| `MOVE_TO` | Rename destination | bright green |
| `MODIFY` | File contents changed | red |
| `WRITE` | File closed after write | red |
| `ATTRIB` | Permissions / ownership changed | yellow |
| `OPEN` | File opened (with `-a`) | dim |
| `READ` | File read (with `-a`) | dim |
| `CLOSE` | File closed without write (with `-a`) | dim |

## How it works

1. **fanotify** — `whodid` calls `fanotify_init(FAN_REPORT_DFID_NAME, …)` to
   open a notification group that embeds file-system object identifiers
   (FIDs) in every event.
2. **Path resolution** — for each event, the directory FID is turned into an
   open file descriptor via `open_by_handle_at(2)`, then the absolute path
   is read from `/proc/self/fd/<n>`.  The file name from the event info
   record is appended.
3. **PID → process** — `readlink("/proc/<pid>/exe")` gives the full
   executable path of the responsible process.
4. **Output sanitisation** — all strings are stripped of control characters
   (including `ESC \x1b`) before printing to prevent terminal
   escape-sequence injection.

---

## Security notes

* `whodid` must run as **root** (or with `CAP_SYS_ADMIN`).  It performs no
  network access, spawns no subprocesses and opens no writable files.
* All path and process-name strings are sanitised before output to prevent
  terminal injection via crafted file names.
* The binary is compiled with `-fstack-protector-strong`, `-D_FORTIFY_SOURCE=2`,
  full RELRO (`-Wl,-z,relro -Wl,-z,now`) and PIE (`-fPIE -pie`).
* Events from `whodid` itself are suppressed by default (use `-s` to include
  them).

---

## WSL2 notes

fanotify works in WSL2 with a kernel ≥ 5.1.  Run `wsl --update` in Windows
PowerShell to update your WSL2 kernel.  WSL1 is **not** supported (it does
not expose fanotify).
